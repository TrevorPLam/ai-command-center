## Pointer Prompt (Short Form)

Read the file at [path/to/tasks/TASK-ID.md] completely before taking any action.

Your entire execution scope is defined by that file. Do not consult any other
document, prior session memory, or assumed project knowledge.

You MAY read files that match ALLOWED_CONTEXT_READS in SECTION 1 of the task
file (e.g., package.json, tsconfig, *.d.ts, MASTER-CONTRACTS.md) to resolve
imports, types, or config. Do NOT read or modify any other files unless they
are listed in FILE_MANIFEST or explicitly allowed in the DISCOVER step.

Begin at PROCEDURE step 1. Execute steps in exact order. Complete this task
fully before reading any other task file. When STATUS is set to DONE, stop and
report the value of NEXT_TASK.

After writing staged files (PROCEDURE step 5), do **not** re‑read the task
spec file. Any bootstrap hash updates are performed by editing the file
directly, and you must rely on the in‑memory state from the initial read.

Do not open, read, or act on any other task file unless explicitly instructed.

## Session Start Prompt (Full Form)

You are executing a task from a multi-module project using isolated task files.

RULES:
1. Read [path/to/tasks/TASK-ID.md] completely before touching any file.

2. Your scope is defined 100% by that file:
   - You MAY read files matching ALLOWED_CONTEXT_READS from SECTION 1 to resolve
     imports, types, and configuration. This now includes MASTER-CONTRACTS.md.
   - You MUST NOT read or modify any other file unless:
       a) It appears in FILE_MANIFEST, or
       b) It is explicitly allowed side-effect in a DISCOVER step.

3. Execute PROCEDURE steps in exact order, one step at a time. Do not reorder,
   merge, or skip steps.

4. If a step is DISCOVER:
   - You MAY inspect logs, test output, and runtime behavior using only commands
     listed in discover.allowed_side_effects for that file.
   - You MUST NOT modify tracked source files in this step.
   - Your output is the structured discovery artifact defined in FILE_MANIFEST.

5. For file writes:
   - All writes MUST go to the staging directory ".agent/staging/[TASK-ID]/"
     during WRITE_STAGED_FILES.
   - Agent performs atomic moves using file operations (move/rename)
     during COMMIT_STAGED_FILES.
   - For PATCH operations: verify hash, apply patch, and re‑hash.
   - If a file has bootstrap.enabled true and expected_sha256 PENDING, after
     writing compute its hash and directly update the task spec file (outside
     staging) to set the hash and disable bootstrap. You must NOT re‑read the
     task file after this update.

6. If a step fails:
   - Use the appropriate SELF_HEALING_RECIPES from SECTION 7.
   - Agent parses error output to classify and select appropriate fix.
   - Respect retry_limit and do not exceed it.

7. If self-healing fails after the allowed retries:
   - Execute ON_UNRECOVERABLE_FAILURE from SECTION 10.
   - Write the failure report and release all locks.
   - Do NOT proceed to any further steps or to the NEXT_TASK.

8. When all VERIFICATION commands in SECTION 9 pass and all ON_COMPLETE steps in
   SECTION 10 succeed, report:
   "TASK [TASK-ID] COMPLETE. Next task: [NEXT-TASK-ID]."

9. After completion:
   - Do NOT open, read, or act on the next task file.
   - Stop and wait for explicit instruction to continue with the reported NEXT_TASK.

10. If the task file is malformed or missing required sections:
    - STOP and report: "TASK FILE MALFORMED: [specific issue]"
    - List missing or malformed sections
    - Do not attempt to proceed with execution
    - Wait for user to fix the task file

Begin now. First action: read [path/to/tasks/TASK-ID.md] in full, then perform
PROCEDURE step 1 (PRE_FLIGHT) exactly as specified.

If the task file cannot be read or is clearly malformed, STOP and report the issue
immediately.
