# Markdown: A Comprehensive Guide to Basics, Best Practices, and Standards

## What is Markdown

At its core, Markdown is a lightweight markup language created in 2004 by John Gruber and Aaron Swartz. It allows users to format text for the web using simple characters--including asterisks, underscores, and brackets--rather than clicking through formatting toolbars. The overriding design goal of Markdown’s formatting syntax is to make it as readable as possible: formatted text should remain perfectly legible both to humans and to programs that convert it into PDFs, websites, or printed books.

Markdown uses formatting symbols to achieve formatting effects. For example, to emphasize text, you simply surround it with `*` or `_` symbols. Markdown has been adopted by a wide range of applications, including Slack, Discord, Microsoft Teams, Notion, GitHub, and Obsidian.

### What Problem Does Markdown Solve?

- **Cross-platform compatibility**: Markdown works across nearly all editors and collaboration tools.
- **Content focus**: It enables greater focus on content creation compared to rich text processors like Word.
- **Version control friendly**: As plain text, Markdown documents can be managed with Git and other version control systems.
- **Future-proof**: Plain text never becomes obsolete.

### Key Benefits of Using Markdown

Formatting text with Markdown is more precise than Word’s rich text formatting while being far simpler to write--and read--than HTML code. Markdown-formatted text maintains consistent appearance when pasted across applications, something that cannot be said for formatted text copied from Word. Two main benefits are the ability to format documents without using a mouse or trackpad and the separation of content from layout.

## Part 1: Markdown Basics

### Headings

Markdown supports six levels of headings, indicated by adding 1 to 6 `#` characters at the beginning of a line. A space must separate the `#` symbol and the heading text.

| Heading Level | Syntax |
|---------------|--------|
| H1 | `# Heading 1` |
| H2 | `## Heading 2` |
| H3 | `### Heading 3` |
| H4 | `#### Heading 4` |
| H5 | `##### Heading 5` |
| H6 | `###### Heading 6` |

### Text Formatting (Emphasis)

| Style | Syntax | Example |
|-------|--------|---------|
| Italic | `*text*` or `_text_` | *italic* |
| Bold | `**text**` or `__text__` | **bold** |
| Bold + Italic | `***text***` or `___text___` | ***bold italic*** |
| Strikethrough | `~~text~~` | ~~strikethrough~~ |
| Inline Code | \`code\` | `code` |

### Line Breaks and Paragraphs

- **Paragraphs**: Separate paragraphs with a blank line.
- **Line breaks**: Add two spaces at the end of a line followed by a newline, or insert a blank line between lines.

### Horizontal Rules

Create a divider by entering three or more asterisks `***` or hyphens `---` on a new line.

### Lists

#### Unordered Lists
Use `-`, `*`, or `+` followed by a space:

```markdown
- First item
- Second item
  - Nested item (indent with spaces or tabs)
- Third item
```

#### Ordered Lists
Use a number followed by a period and a space:

```markdown
1. First item
2. Second item
3. Third item
```

#### Task Lists (Checklists)
Use `- [ ]` for an incomplete task or `- [x]` for a completed task.

### Links

Two forms of links exist: **inline** and **reference-style**.

**Inline links**: `[Link text](URL)`

**Reference-style links**:
```markdown
[Link text][label]
[label]: https://example.com
```
Reference-style links allow defining a link destination once and reusing the label multiple times throughout a document.

**Raw URLs**: Enclose the URL in angle brackets: `<https://example.com>`

### Images

`![Alt text](image.jpg)`

The alt text appears when the image cannot be loaded and improves accessibility.

### Blockquotes

Start each line of the quote with `>` followed by a space:

```markdown
> This is a blockquote.
> It can span multiple lines.
```

### Code Blocks

**Inline code**: Wrap code in single backticks: `` `code` ``

**Fenced code blocks**: Wrap code lines with three backticks (```) before and after the block. Blank lines above and below the code block should be added. Syntax highlighting can be specified by adding the language name after the opening backticks.

### Tables

Basic table syntax uses pipes (`|`) to separate columns and hyphens (`-`) to define the separator between the header and the table body:

Left-Aligned Column | Center-Aligned Column | Right-Aligned Column
:--- | :---: | ---:
Data | is | here

**Alignment syntax**: Colons control column alignment:
- `:---` = left-aligned
- `:---:` = centered
- `---:` = right-aligned

Leading and trailing pipes in table rows are optional, but including them improves readability.

## Part 2: Markdown Fundamentals

### Core Design Philosophy

Markdown’s fundamental design principle is **readability first**. The syntax should remain legible as plain text even before conversion. The language separates content from presentation--authors focus on writing while the rendering engine handles the appearance.

### Block Structure vs. Inline Structure

Markdown parsing follows a two-phase approach:

1. **Block Structure Phase**: Each line is analyzed for block-level markers. Container blocks (blockquotes, lists) are identified, leaf blocks (headings, code blocks, paragraphs) are processed, and link reference definitions are collected.
2. **Inline Structure Phase**: Inline formatting (emphasis, links, code spans) is processed within the identified blocks.

This two-phase design enables consistent and predictable parsing across implementations.

### Whitespace Rules

Whitespace serves as a signal for block boundaries in Markdown:
- Place a **single blank line** between Markdown blocks of different types (e.g., between a paragraph and a list or between a paragraph and a heading).
- Do not use more than one blank line between blocks; multiple blank lines render as a single blank line in HTML and are therefore unnecessary.
- Do not use multiple consecutive blank lines inside a code block, as this will break the code block.
- Remove extra spaces at the end of lines, as trailing spaces can change how Markdown renders.
- Always use spaces instead of tabs (hard tabs).

### Character Escaping

To display a character that Markdown would normally interpret (such as `#`, `*`, or `_`), precede it with a backslash (`\`). This escapes the character’s special meaning and displays it literally.

### What Markdown Is Not

If Markdown lacks a specific formatting need, fallback options exist. Missing elements can be handled by dropping in a few LaTeX commands or raw HTML tags, as Markdown supports inline HTML.

## Part 3: Markdown Best Practices

### General Formatting Guidelines

| Practice | Guideline |
|----------|-----------|
| Blank lines | Use exactly one blank line between different block types |
| Blank lines (max) | Never use more than one consecutive blank line |
| Trailing spaces | Remove all trailing spaces at line ends |
| Tabs vs. spaces | Always use spaces, never hard tabs |
| Line length (general) | Limit to 100 characters for most content types |
| Line length (about_ articles) | Limit to 79 characters |
| File endings | End each file with a single empty newline |

**Source**: Microsoft PowerShell Documentation Best Practices

Limiting line length improves the readability of git diffs and history and makes it easier for other writers to contribute. The [Reflow Markdown](https://marketplace.visualstudio.com/items?itemName=marvhen.reflow-markdown) VS Code extension can reflow paragraphs to fit the prescribed line length.

### Headings Best Practices

- **Use ATX headings only**: Use the `#` style headings rather than underlined `=` or `-` style headers.
- **Title case**: Use sentence case--only proper nouns and the first letter of a title or heading should be capitalized.
- **Single H1 per document**: Include only one H1 per document. It should be the first heading and should match the article title.
- **Space after #**: Include a single space between the `#` and the heading text.
- **Blank lines around headings**: Surround headings with a single blank line.
- **Do not skip levels**: Increment heading levels by exactly one at a time. After an H1 (`#`), the next heading must be H2 (`##`), not H3 (`###`).
- **Limit depth**: Restrict heading depth to H3 or H4 when possible.
- **Avoid bold or markup in headings**: Do not use bold, italics, or other inline markup inside headings.
- **Avoid stacked headings**: Never follow a heading immediately with another heading without content in between.
- **Avoid duplicate headings**: If reusing a heading, keep its meaning consistent to avoid confusing users.

### Text Emphasis (Bold & Italic) Best Practices

**General practices**:
- Use underscores (`_`) for italics and double asterisks (`**`) for bold. This makes it easier for humans to distinguish italics from bold in the source Markdown file.
- Use bold formatting only for UI elements and the beginning of notices (e.g., Important, Warning).
- Use italics sparingly. Use underscores for italics when drawing attention to a specific word or phrase, defining terms, or using words as words.
- Italicize titles of full-length works (books, movies, web series) unless they are part of a link.
- Italicize mathematical variables and version variables (e.g., `x + y = 3`, `version 1.4.x`).
- Do not use underlining except for link text.

### Links Best Practices

- **Prefer inline links over raw URLs**: Use `[Link text](URL)` instead of bare URLs to improve readability.
- **Use reference-style links for repeated destinations**: Define the link once and reuse it multiple times to simplify maintenance.
- **Add meaningful link text**: Link text should describe or indicate the destination, not just say “click here”.
- **Validate links**: Ensure links are functional and point to correct destinations.

### Lists Best Practices

- **Maintain consistent indentation**: Sublists should be indented consistently (typically 4 spaces).
- **Use blank lines appropriately**: Place a blank line before the start of a list for proper parsing.
- **Keep lists parallel**: In ordered lists, ensure items follow a parallel grammatical structure.
- **Use task lists for checklists**: Prefer `- [ ]` syntax for checklists rather than bullet points.

### Code Blocks Best Practices

- **Use fenced code blocks for multi-line code**: Use three backticks (` ``` `) rather than indented blocks.
- **Specify language for syntax highlighting**: Add the language name after the opening backticks (e.g., ` ```python `).
- **Add blank lines around code blocks**: Insert blank lines above and below code blocks for proper spacing.
- **Use inline code markers**: Use single backticks for inline code elements (filenames, class names, method names, HTTP status codes) rather than styling manually.

### Tables Best Practices

- **Include leading and trailing pipes**: Although optional in GFM, including leading and trailing pipes for table rows improves readability and prevents parsing ambiguity.
- **Use consistent column counts**: Ensure every row has the same number of columns.
- **Align data meaningfully**: Use colons to control column alignment appropriately for the content type (left for text, right or center for numbers).

### File and Folder Naming Best Practices

- Markdown filenames must be **lowercase with hyphens**. No capital letters, underscores, periods, or spaces. (Image assets can include capital letters and underscores, though not recommended).
- Avoid filenames that can conflict with JavaScript/CSS conventions, such as `metadata.md` and `search.md`.
- Avoid filenames that conflict with reserved keywords (`search`, `results`, `facets`, `pagination`, `sorting`, `query`, `searchbox`, `content`, `header`, `footer`, `main`, `navigation`, `sidebar`, `page`, `container`, `wrapper`).

### Heading Anchors (IDs) Best Practices

- Heading IDs should not include periods. For version numbers, use hyphens instead: `{#version-2-3}` rather than `{#version-2.3}`.
- Every section heading referenced in a table of contents must have a valid section ID.
- No duplicate heading IDs are allowed within a single document.

## Part 4: Highest Standards -- CommonMark and Markdown Specifications

### The Problem CommonMark Solves

The original Markdown specification by John Gruber was intentionally ambiguous in many areas, leading to different interpretations by various implementations. Consequently, the same Markdown document would render differently on different platforms (GitHub, StackOverflow, Reddit, and others).

CommonMark was created to address these ambiguities and inconsistencies. It is a **strongly specified, highly compatible implementation of Markdown**.

### What CommonMark Provides

- **Unambiguous specifications** for all Markdown syntax
- **Comprehensive test suite** to ensure consistent behavior across implementations
- **Clear precedence rules** for conflicting syntax
- **Detailed parsing algorithm** that can be implemented consistently

### Key Differences Between Standard Markdown and CommonMark

| Element | Standard Markdown | CommonMark |
|---------|-------------------|------------|
| **Blank lines before block elements** | Often allows headings/blockquotes/lists without blank lines | Requires blank lines before these elements |
| **List item indentation** | Varies by implementation | Specific rules: sublists require consistent indentation (typically 4 spaces) |
| **Fenced code blocks** | Inconsistent behavior | Standardizes syntax with backticks or tildes; requires consistent indentation and closing markers |
| **Indented code blocks** | Often allowed without blank lines | Requires blank lines before indented code blocks |
| **Link reference precedence** | Varies | Clear rules for which reference definition takes precedence when multiple definitions exist |
| **Nested emphasis** | Ambiguous parsing | Uses a “delimiter stack” algorithm for consistent emphasis parsing |
| **Line breaks** | Varies | Requires two spaces at end of line for hard breaks; single line breaks become soft breaks |
| **HTML blocks** | Varies | Defines 7 types of HTML blocks with specific start/end rules |

**CommonMark parsing algorithm**:
1. **Phase 1 -- Block Structure**: Lines are analyzed for block-level markers. Container blocks (blockquotes, lists) are identified, leaf blocks (headings, code blocks, paragraphs) are processed, and link reference definitions are collected.
2. **Phase 2 -- Inline Structure**: Inline formatting is processed within the identified blocks.

### CommonMark in Practice

The CommonMark specification defines a rationalized version of Markdown syntax, accompanied by a specification and BSD-licensed reference implementations in C and JavaScript. As of January 2025, the current specification version is **0.31.2** (released January 28, 2024).

Many documentation pipelines (including Microsoft’s Learn platform and Grafana’s documentation builds) use CommonMark-compliant parsers such as **markdig** to process Markdown documents. Goldmark (Hugo’s Markdown parser) also supports the CommonMark flavor.

### GitHub Flavored Markdown (GFM)

GFM is GitHub’s extension of the CommonMark specification. It builds on CommonMark as a baseline and adds platform-specific extensions.

**Extensions introduced by GFM include**:
- Tables (extended from basic Markdown)
- Task lists (`- [ ]`)
- Strikethrough (`~~text~~`)
- Autolinks for URLs
- Emoji shortcodes (`:smile:`)
- Syntax highlighting for fenced code blocks

**Key GFM best practices**:
- Use real headings (not bold lines) for document structure and screen reader navigation.
- Do not rely on color alone; use text labels and icons.
- Keep line length readable (~80-100 characters) for nicer diffs in version control.
- When using GFM table syntax, include leading and trailing pipes for readability, even though they are optional.

Use GFM Markdown syntax whenever possible, and fall back to raw HTML when necessary or when HTML is more readable.

## Part 5: Markdown Extended Syntax and Advanced Features

Many platforms and renderers extend standard Markdown/CommonMark with additional features.

### Common Markdown Extensions

| Extension | Description |
|-----------|-------------|
| **Admonitions/ Alerts** | GitHub-style callouts (Note, Tip, Info, Important, Warning, Caution, Danger) with customizable titles |
| **Math Rendering** | Inline (`$...$`) and display (`$$...$$`) math equations using KaTeX or MathJax |
| **Mermaid Diagrams** | Embedded diagram rendering for flowcharts, sequence diagrams, and other visualizations using Mermaid.js |
| **Tables** | Full table support with alignment options and advanced features (MUI DataGrid, etc.) |
| **Footnotes** | Reference-style footnotes with automatic numbering and back-references |
| **Definition Lists** | Term-definition pairs with colon syntax |
| **Spoilers/ Collapsible Sections** | Hidden content (e.g., `||hidden text||`) or `<details>`/`<summary>` HTML elements for collapsible sections |
| **Subscript and Superscript** | Subscript (`~text~`) and superscript (`^text^`) formatting |
| **Highlighting** | Marked/highlighted text (`==highlight==`) |
| **Inserted (Underline)** | Inserted text (`++underline++`) |
| **Emoji** | Shortcode support (`:emoji_name:`) for emoji insertion |

### Image Dimensions

Some extended versions of Markdown allow specifying image dimensions by adding an argument `=/height/x/width/` to the link description.

### When to Use Extended Syntax

- Use extended syntax **when you control the rendering environment** and can guarantee support.
- Be aware that extended syntax is **tool-specific** or **platform-specific**; not every Markdown parser will support it.
- For maximum portability, rely on **CommonMark** as the baseline and treat extended features as optional enhancements.
- When in doubt about cross-platform support, fall back to **standardized CommonMark** or raw HTML.

### Performance Awareness

Rendering Markdown in dynamic applications (such as React) triggers a full parse on every re-render. As documents grow, performance can degrade. For production applications, consider implementing lazy-loading strategies for computationally expensive features like syntax highlighting and diagram rendering.

## Part 6: Tools for Enforcing Markdown Quality

### markdownlint

**markdownlint** is a static analysis tool for Node.js with a library of rules to enforce standards and consistency for Markdown files.

**Key markdownlint rules**:

| Rule ID | Description |
|---------|-------------|
| MD001 | Heading levels should only increment by one level at a time |
| MD025 | Only one H1 heading is allowed per document |
| MD030 | Spaces after list markers -- typically one space after the marker |

**How to use**:
- **VS Code extension**: Install the markdownlint extension (included in the Learn Authoring Pack for Microsoft Learn contributors).
- **Command-line interface**: Use `markdownlint-cli` for bulk linting, which can be integrated into build pipelines for automated testing.
- **Configuration**: `.markdownlint.yaml` files can extend shared configurations and optionally disable rules considered too pedantic for specific projects.

### markdig

Microsoft’s documentation build pipeline uses **markdig** to process Markdown documents. markdig parses documents based on the rules of the latest CommonMark specification and adds extensions for platform-specific features such as tables and alerts.

### Additional Recommended Tools

- **VS Code extensions**: The Learn Authoring Pack (includes markdownlint and Reflow Markdown for line length management)
- **Prettier**: Can be combined with markdownlint for consistent formatting across the entire project
- **Adobe Markdown Authoring extension**: Available in VS Code Marketplace for enhanced authoring features

## Summary: Quick Reference of Highest Standards

| Standard | Description |
|----------|-------------|
| **CommonMark** | The unambiguous Markdown specification that ensures consistent rendering across all CommonMark-compliant parsers. Current version: 0.31.2. |
| **GitHub Flavored Markdown (GFM)** | The standard for GitHub documentation, extending CommonMark with tables, task lists, strikethrough, and other features. |
| **One H1 per document** | Highest industry standard: every document should have exactly one top-level heading. |
| **One blank line between block types** | The recommended separator for paragraphs, lists, headings, code blocks, and tables. |
| **100-character line limit** | The most widely adopted maximum line length for Markdown source code in production documentation systems. |
| **Spaces, never tabs** | Use spaces for indentation to ensure consistent rendering across editors and parsers. |
| **No trailing spaces** | Trailing spaces can change rendering behavior unpredictably; always strip them. |
| **UTF-8 encoded files** | Use UTF-8 encoding for full character set support and cross-platform compatibility. |
| **ATX headings only** | Use `#` style headings, not underlined `=` or `-` styles. |
| **Sentence case for headings** | Only proper nouns and the first letter should be capitalized. |

### Final Recommendations

1. **Use CommonMark as your baseline** to ensure maximum compatibility across platforms.
2. **Adopt markdownlint** (or equivalent linting tool) to enforce consistency automatically.
3. **Apply the one-blank-line rule** consistently between all block types.
4. **Use spaces (never tabs)** and enforce a 100-character line limit.
5. **Keep headings logical and semantic** -- one H1, increment by one level, and never stack headings without content.
6. **Include leading and trailing pipes in tables** for maximum readability.
7. **Specify language identifiers for code blocks** to enable syntax highlighting.
8. **Validate all links** before publishing or committing.
9. **Avoid extended syntax** unless you are certain of the rendering environment’s capabilities.
10. **End every file with a single newline character** to prevent diffs and tool-specific issues.

Markdown’s strength lies in its simplicity and readability. The highest standard for Markdown is the one that makes your content both **easy to write** and **reliably consistent** across all destinations where it will be consumed. By adhering to CommonMark specifications and maintaining disciplined formatting practices, you ensure that your documentation remains maintainable, accessible, and professional.
